「コンテキスト」って何？
🧠 AIが「今見えている情報」の全体

AIが回答を生成するとき、参照できる情報には 「コンテキストウィンドウ」 という上限があります。その中に入っているすべての情報を コンテキスト と呼びます。

📦 コンテキストウィンドウの中身
⚙️
システムプロンプト
「あなたはカスタマーサポートです」のような基本指示
🔧
ツール定義

📚
外部データ・RAG
データベースや資料から取得した情報
💬
会話履歴
これまでのやりとり
📋

AIのコンテキストウィンドウは、試験の「持ち込み許可のカンニングペーパー」みたいなもの。
スペースが決まっているので、何を書くか（＝何を渡すか）をうまく選ぶ必要があります。

なぜ重要？「コンテキスト劣化」問題
📉 情報が増えると、AIの精度が落ちる

コンテキストに詰め込む情報が多いほど、AIは「コンテキスト劣化（Context Rot）」と呼ばれる現象を起こし、肝心な情報を見落とすようになります。

コンテキストが少ない（理想）
集中力 85%
余裕あり
コンテキストが多い（問題あり）
集中力 35%
情報の海に溺れている…
人間でも「机の上に書類が山積みだと、肝心な資料を見つけられない」のと同じ。
AIには 「注意力の予算（Attention Budget）」 があり、情報が増えるほど一つひとつに割ける集中力が減ります。

技術的には、Transformerアーキテクチャはトークン数 n に対して n² の関係 を計算するため、長くなると指数的に負荷が増えます。

良いコンテキストの作り方
🎯 黄金ルール：「小さくて・意味のある」情報だけ渡す
"find the smallest possible set of high-signal tokens that maximize the likelihood of some desired outcome"
→ 求める結果を最も引き出せる、できるだけ少ない「意味のある情報」を選べ。

① システムプロンプトの「高度（Altitude）」を合わせる

❌
細かすぎ
「もし〜なら、〜して、〜でなければ…」と if-else で埋め尽くす → 壊れやすい
✅
ちょうどよい
具体的すぎず、曖昧すぎず。AIが自分で判断できる余白を残す
❌
曖昧すぎ
「いい感じにやって」— 共通認識を勝手に仮定している → 迷子になる

② トークンの「価値」で仕分けする

✓ 目標・ゴール ✓ 具体的なルール ✓ 良いサンプル例 △ ツールの説明 △ 必要な会話履歴 ✗ 重複した情報 ✗ 古いツール実行結果 ✗ 使わないサンプル

③ ツールは「少なく・明確に」

ツールが多すぎると「どれを使えばいいか」AIが迷います。人間のエンジニアが「この状況でどのツールを使うか」を即答できるレベルまで整理しましょう。
5
「必要になったら取得」Just-in-Time 戦略
📂 事前に全部渡さない。参照先だけ持っておく

RAG（事前検索して情報を埋め込む）に加え、最近は AIエージェント自身が実行中に必要な情報を「ちょうどいいタイミングで」取りに行く方法が主流になっています。

軽量な
参照情報だけ保持
ファイルパス・URL
→
必要に応じて
ツールで取得
glob / grep / API
→
必要な部分だけ
コンテキストへ
最小限の情報
「教科書を丸ごと暗記する」のではなく、「索引と本棚の場所を覚えておいて、必要なページだけ開く」 イメージ。
Claude Code（Anthropicのコーディングエージェント）はこの方式で巨大なコードベースを分析しています。
ハイブリッド戦略： 速さが必要な情報は事前取得（RAG）、深掘りが必要な情報はエージェントが自律探索。 両方を組み合わせるのが現在のベストプラクティスです。
6
長期タスクを乗り越える3つのテクニック

何時間もかかるタスク（大規模コード移行・調査プロジェクトなど）では、コンテキストウィンドウが溢れる問題が起きます。それを解決する3つの手法があります。

🗜️

① コンパクション
Compaction

会話が長くなったら、AIに「ここまでの要点をまとめて」と頼んで、新しい短い会話として再スタート。重要な決定事項・バグ・実装の詳細だけ残し、不要なログは捨てる。

📓

② 構造化メモ書き
Structured Note-taking

AIが自分で「NOTES.md」や「TODO.md」に進捗・学習・方針を書いておき、コンテキストがリセットされても読み返せる「外部記憶」として活用する。

🤝

③ マルチエージェント
Sub-agent Architecture

1つのAIが全部やらず、専門AIを複数起動。各サブエージェントが深掘りして「1,000〜2,000トークンの要約」だけ親AIに返す。分業でコンテキストを節約。

使い分けの目安

🗜️ コンパクション — 会話が長くなる双方向タスク（チャットボット・対話型作業）
📓 メモ書き — マイルストーンが明確な反復開発（機能実装・テスト）
🤝 マルチエージェント — 並列調査が活きる複雑な研究・分析

🎓 まとめ：新卒エンジニアへのメッセージ

「AIに渡す情報を、
どう選ぶかが、
エンジニアリングの新しい腕の見せどころ」

AIエージェントを使う現場では「プロンプトを上手く書く」だけでは不十分になってきました。
コンテキストウィンドウという有限なリソースをどう設計・管理するかが、 AI活用の品質を左右します。

コンテキストエンジニアリングの3原則：
① 情報は小さく・意味のあるものだけ渡す
② 必要な情報は都度取得させる
③ 長期タスクは「圧縮・メモ・分業」で乗り越える
